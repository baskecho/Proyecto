REST

  • Estilo de arquitectura
  • Nació de la tesis de doctorado de Roy Fielding en el 2000
    • desarrollo http 1.1 entre 1996-1999
  • Basado en recursos en contraste de basado en acciones
    • Le da mayor peso al recurso
  • Apache
    • contenido estático de una página web
  • Http
    • Es un protocolo de aplicación
    • Caché de navegador es una memoria temporal que tiene una vigencia
      • Apache
        • Contenido estático de una página web
        • a todo el contenido estático le podemos hacer caché
      • Con f5 borro el cache y actualice la página
    • para sistemas de información distribuidos, colaborativos hipermedia
    • 2018 se promovió protocolo http 3
    • Métodos para crear el CRUD
      • GET
        • Es una petición al servidor o recurso por ser REST
        • Debe ser idempotente
        • El get se escribe en el barra de direcciones.
          • URI - URL
          • cada vez que escribo una dirección estoy haciendo un GET
      • HEAD
      • POST
        • para crear el get
        • no es idempotente
        • summit a un formulario le estoy haciendo un POST
        • enviar una contraseña esta dentro del POST
      • PUT
        • actualizar en bloque
      • PATCH
        • para las diferencias
      • DELETE
        • Si es idempotente
        • si lo borro en la base de dato, cuando lo vuelva a borrar sale error por lo que no es idempotente, pero para volverlo idempotente se marca como anulada la tabla, no se elimina la información, una tra
      • CONNECT
      • OPTIONS
      • TRACE
    • Idempotente
      • devuelve el mismo resultado
    • Si la función está cambiando, está mal implementada
  • Tiene seis restricciones, uno opcional, los otros obligatorios, se tiene que hacer los primeros 5
    • uniform interface
      • Define la interface entre el cliente y el servidor
      • Simplifica y desacopla la arquitectura = evoluciona diferentemente (back,front)
      • Fundamentalmente diseño RESTfull
      • Significa
      • Pensar en definir un API
        • permitió separar las responsabilidades del servidor y cliente
      • Recurse-based
        • Basados en cosas y no en acciones
        • un mismo recurso puede implementar varios métodos
        • nombres vs verbos
        • Versus SOAP-RPC
        • Identificados por URIs
          • múltiples URIs pueden apuntar al mismo recurso
          • Cuando se tiene un modelo complejo como el del proyecto se puede cometer el error carros,pilotos, mecánicos, repuestos, URL muy compleja, se puede hacer que múltiples apis apunten al mismo recurso
        • Separar la representación de los recursos.
        • antes se conocía como paradigma/documento/vista
          • no se podía separar los datos de la representación, pero hoy en dia se puede hacer.
          • Ahora La misma información base se puede convertir en diferentes formas
      • Manipulation resources through representation
        • como los recursos son manipulados.
        • transferencia entre el cliente y el servidor
        • Típicamente JSON/XML
          • Json más común porque es más liviano y tiene menos texto
          • XML es más estructural y mejor formado
        • Ejemplo
          • Recurso: persona(Alejo)
          • Servicio: información contacto (GET)
          • Representación
            • Nombre, dirección, celular
            • JSON, XML
      • selft-descriptive messages
        • Cada mensaje incluye información para describir cómo procesar el mensaje
      • typemedia as the application state (HATEOAS)
        • Dentro de la respuesta mandamos el enlace
        • se entrega el estado:
          • clientes
            • body
            • query
          • Servidor
            • body
            • responsive
    • Stateless
      • El servidor no guarda el estado del cliente
        • el servidor no sabe quien consultó la lista
        • cookies guardan el estado de los cliente
          • se están volviendo un hueco de seguridad
      • Cada REST contiene suficiente contexto
        • mensajes autodestructivos
      • cualquier información del estado es guardada en cliente
    • Cacheable
      • La respuesta del servidor se puede guardar en cache para acelerar
        • implícitamente
        • explícitamente
        • negociable
      • Si se puede guardar en cache un recurso se puede usar sin necesidad de ir al servidor.
    • Cliente-servidor
      • separación de responsabilidades
    • Sistema en capas
      • El cliente no asume conexión directa con el servidor, separación de responsabilidades
      • Hay intermediarios de HW y SW
        • Por ejemplo: balanceadores de carga
      • Mejora la escalabilidad
    • Código por demanda
      • ultima y unica opcion opcional
      • El servidor puede temporalmente extender al cliente
      • se transfiere lógica al cliente

CONCLUSIONES

  • Aplicaciones que cumplan con las 5 restricciones cumple beneficios como
    • Portabilidad
    • Escalabilidad
    • Confiabilidad
    • Simplicidad
    • Visibilidad
  • Crecimiento vertical
    • subir o reducir capacidad como por ejemplo en AWS aumentar la capacidad
    • Ejemplo
      • e3 micro 1 VCPU 1 GB que soporta 100 usuario
      • e3 micro 10 vCPU 10 GB soporta 1000 usuarios
      • Al crecer la máquina la probabilidad de fallo también aumento, se cae y se va con los clientes concurrentes que en ese momento estaban.
  • Crecimiento Horizontal
    • se podría borrar la sesión a los tomcast y pasarla a la base de datos, pero ahora todo el trabajo la tiene las bases de datos, pero se podría partir o crear varias bases de datos
      • Teorema del CAP
        • Es imposible que un sistema de cómputo distribuido satisfaga:
          • consistencia
          • Disponibilidad
          • Tolerancia al particionado.
  • Usar nombres en vez de verbos
    • Verbos
      • Crear cast
      • ver cast
      • abrir cast
    • Nombres y en plural
      • GET/users/711/active
      • /car /cars
      • /user /users
      • /product /products
    • Relaciones
      • GET /cars/711/drivers
        • devuelve una lista de conductores para carro 711
      • GET/cars/711/drivers/4
  • API
    • métodos
    • PUT
    • POST
  • Usar HATEOAS
    • Es un principio para que los links de hipertexto sean usados para navegar a través de API
  • Filtros
      data-aos="fade-up" data-aos-duration="1000" data-aos-delay="0"
    • Usar un único parámetro o lenguaje
    • Se filtrar por un único parámetro o se utiliza un lenguaje como JQUERY
  • Ordenamiento
  • Selección de los campos
    • Selección de las columnas, al traer información de un recurso
      • GET /cars?fields=manufacturer,model,id,color
  • Paginación
    • Usar limit y offset, es flexible y está disponible en varias bases de datos, los valores por omision limmit=20 offset=0
    • GET /cars?offset=10&limits=5
  • Versionar el API
    • Hacer el versionamiento del API obligatorio, preferiblemente en números ordinales
  • Manejo de errores
    • Tipos
      • 1xx - información
      • 2xx - Éxito
      • 3xx - Redirección
      • 4xx - Error en el cliente
      • 5xx -Error en el servidor
    • Ejemplo
      • 200 - OK -todo esta trabajando
      • 201 - OK - nuevo recurso fue creado.
      • 401 - donde no se tiene permiso
      • 403 mo estar metido aquí
      • 404 - MUY COMÚN
      • 422 -
      • 500 - Internal server Error - no manejarlo dejar que el servidor lo haga.
